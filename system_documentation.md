# Science Week Web System - Comprehensive Documentation

เอกสารนี้รวบรวมรายละเอียดทั้งหมดของระบบ "สัปดาห์วิทยาศาสตร์ โรงเรียนวัชรวิทยา" รวมถึงสถาปัตยกรรม (Architecture) โครงสร้างฐานข้อมูล และวิธีการทำงานของระบบแต่ละส่วน

---

## 1. System Architecture (สถาปัตยกรรมระบบ)
ระบบนี้ถูกออกแบบให้เป็น **Serverless Web Application** โดยมีการแยกส่วน Frontend และ Backend อย่างชัดเจน และไม่มีการเช่า Server หรือ Database เพิ่มเติมใดๆ

- **Frontend:**
  - สร้างด้วย HTML, Vanilla JavaScript และ CSS
  - ใช้งาน **Tailwind CSS** (ผ่าน CDN) ในการออกแบบ UI ให้มีความทันสมัยและ Responsive (รองรับมือถือและคอมพิวเตอร์)
  - ใช้งาน **SweetAlert2** สำหรับการแสดง Popup แจ้งเตือน และสถานะการโหลดต่างๆ

- **Backend / API:**
  - ใช้ **Google Apps Script** (`code.gs`) เป็นตัวกลาง (API) สำหรับการรับ-ส่งข้อมูลผ่าน HTTP (GET / POST)
  - ผู้ใช้ไม่จำเป็นต้องจัดการเซิร์ฟเวอร์ด้วยตนเอง
  - สร้าง Web App Deployment เพื่อใช้เป็น API URL

- **Database:**
  - ใช้ **Google Sheets** เป็นฐานข้อมูลหลักสำหรับเก็บข้อมูล (ข่าวสาร, กำหนดการ, รายการแข่งขัน, ข้อมูลผู้สมัคร, และการส่งผลงาน)

- **Storage:**
  - ใช้ **Google Drive** สำหรับรับไฟล์ที่นักเรียนอัปโหลดตอนส่งผลงาน (ผ่านการส่ง Base64 และสร้างไฟล์ผ่าน Apps Script)

## 3. Workflow และ Logic สำคัญ

### 3.1 ระบบการสมัครแข่งขัน (Registration)
1. **Frontend (`register.html`)**: โหลดรายชื่อการแข่งขันที่สถานะ `register_open = true` มาให้เลือก
2. ระบบตรวจสอบว่าเป็นประเภทเดี่ยว (`team_max = 1`) หรือประเภททีม และปรับ UI ตาม
3. **Backend (`code.gs` - registerCompetition)**:
   - ตรวจสอบ ID รหัสล่าสุดของแถว แล้วสร้างรหัส `REGxxx` ใหม่
   - บันทึกเบอร์โทรศัพท์โดยเติมเครื่องหมาย `'` ด้านหน้า เพื่อกันไม่ให้ Google Sheet แปลงเป็นตัวเลขแล้วตัดเลข 0 ออก
   - เพิ่มข้อมูลทั้งหมดลงในชีต `registrations`

### 3.2 ระบบการส่งผลงาน (Submission)
1. **Frontend (`submit.html`)**: ให้ผู้ใช้งานกรอกรหัสการสมัคร และแนบไฟล์ผลงาน (จำกัดขนาดตามเบราว์เซอร์)
2. มีการใช้ `SweetAlert2` เด้งขึ้นมาบอกสถานะ "กำลังอัปโหลดไฟล์..." และบล็อกการกดปุ่มซ้ำ
3. เมื่อแนบไฟล์ Frontend จะทำการแปลงไฟล์เป็น `Base64` เพื่อส่งผ่าน HTTP POST แบบ JSON ไปยัง Backend
4. **Backend (`code.gs` - submitWork)**:
   - **Validation 1**: ตรวจสอบว่ามี `registration_code` นี้อยู่ในชีต `registrations` หรือไม่ (ถ้าไม่พบ -> แจ้ง Error)
   - **Validation 2**: ตรวจสอบว่า `competition_id` ตรงกับที่เลือกลงทะเบียนไว้หรือไม่ (ถ้าไม่ตรง -> แจ้ง Error)
   - **Validation 3**: ตรวจสอบในชีต `submissions` ว่ามีคนส่งงานด้วยรหัสการสมัครนี้แล้วหรือยัง (เพื่อกันส่งซ้ำ)
   - **Drive Integration**: เปลี่ยนชื่อไฟล์ที่อัปโหลดใหม่ให้เป็น `[registration_code].[extension]` เช่น `REG001.pdf` แล้วบันทึกลงใน Folder ID ที่ระบุไว้
   - **Sharing Handling**: ข้าม Error กรณีที่ Google Workspace for Education บล็อกการตั้งค่า Public Share ไว้ (`try...catch`) และเก็บลิงก์ URL
   - **Database**: บันทึกข้อมูลทั้งหมดลงชีต `submissions` โดยกำหนดสถานะตั้งต้นเป็น **"ส่งแล้ว"**

### 3.3 ระบบหลังบ้าน (Admin Dashboard)
- มีการใช้ระบบ `adminLogin` แบ่ง Role 
  - `admin` (จัดการได้ทุกอย่าง ตั้งค่าหน้าเว็บ จัดการข่าว ประกาศ)
  - `teacher` (คุณครู - จัดการเฉพาะการเปลี่ยนสถานะรับสมัคร/การส่งงาน แก้ไขข้อมูลสมัคร และดูข้อมูลผลงานเท่านั้น)
- โหลดข้อมูลจาก Sheets ทั้ง 5 ตารางมาแสดงในรูปแบบตารางและป้ายกำกับต่างๆ (UI Tab)
- วันเวลาในส่วน **วันที่ส่ง** แสดงในรูปแบบ **ภาษาไทย** เช่น `15 ก.ค. 2569 เวลา 14:30 น.`

## 4. การจัดการความปลอดภัยและ Error
- ระบบใช้การส่งข้อมูล `POST` ผ่าน API 
- มีการจัดการข้อผิดพลาด (Exception Handling) ทั้งฝั่ง Frontend (`try...catch` + SweetAlert) และฝั่ง Backend
- ข้อมูล Password และโฟลเดอร์รหัส ถูกเก็บเป็นค่าตัวแปรค่าคงที่ใน `code.gs` ทำให้ฝั่ง Frontend (ผู้ใช้งานทั่วไป) ไม่สามารถมองเห็นหรือเข้าถึงได้

## 5. การตั้งค่าและการ Deploy สู่ระบบจริง
ทุกครั้งที่มีการแก้โค้ดในฝั่ง Backend (`code.gs`) จะต้องทำตามขั้นตอนดังนี้เพื่อให้การตั้งค่ามีผล:
1. ใช้งานผ่านหน้าเว็บ Google Apps Script
2. เลือกปุ่ม **การทำให้ใช้งานได้ (Deploy) > การทำให้ใช้งานได้รายการใหม่ (New deployment)**
3. ตั้งค่าเป็น **ประเภทแอปเว็บ (Web App)**
4. ตั้งค่า **ดำเนินการในฐานะ (Execute as)** ให้เป็น `"ฉัน" (Me)` **(สำคัญมาก เพื่อให้สคริปต์เข้าถึงสิทธิ์อัปโหลดไฟล์ใน Drive ได้)**
5. ตั้งค่า **ผู้มีสิทธิ์เข้าถึง (Who has access)** ให้เป็น `"ทุกคน" (Anyone)`
6. นำ URL ที่ได้มา ไปอัปเดตในไฟล์ `assets/js/config.js` ในตัวแปร `API_URL`
